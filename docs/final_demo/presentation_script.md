## Slide 3 · 项目概述 — 侯乔岳

简单介绍一下这个项目。

校园互助服务平台，一句话说，就是让同学们可以发布需求、接单、沟通、评价，形成一个带激励机制的闭环。我们覆盖了六种需求类型：跑腿代取、二手交易、组队匹配、失物招领、学习互助，还有一个开放的「其他」类型。

项目规模上，我们设计了 15 张数据表，实现了 57 个 REST API 和 18 条前端路由，编写了 243 个测试用例，设计了 9 种成就徽章。

技术栈方面，后端用了 Spring Boot 3.2.5 加 MyBatis-Plus，用 Spring Security 加 JWT 做鉴权，WebSocket STOMP 做实时通信，Flyway 管理数据库版本。前端是 Vue 3 加 Vite 5，移动端用 Vant 4 组件库，状态管理用 Pinia。

右边这些是我们项目的几个技术亮点：JWT 无状态认证、积分不可变账本、WebSocket 实时通信、响应式双视图、悲观锁防超发等等，后面会逐一展开。

## Slide 4 · 过程模型：敏捷开发 — 侯乔岳

接下来说一下我们的开发过程。整个项目我们采用敏捷开发的方式。

整个过程从 P0 团队组建和分工，到 P1 需求分析产出 SRS 和用户故事，到 P2 体系结构设计确定分层架构和 ADR，到 P3 详细设计产出类图和顺序图，最后到 P4 编码测试——前后端同步开发，最终跑通了 243 个测试用例。每一阶段都有文档和可交付物。

## Slide 5 · 需求分析 — 侯乔岳

需求分析层面，我们从三个层次来梳理。

首先是业务需求——我们要解决什么问题？提升校园闲置资源的利用率，降低同学之间互助的交易成本，形成一个可信的积分和评价闭环。这是整个项目的价值基础。

往下是用户需求——用户想要什么？快速发布和筛选需求、安全接单和私信沟通、能看到评价和徽章、收到通知。这些都是从用户视角出发的。

再往下是系统需求——技术上系统需要做到什么？JWT 鉴权和角色控制、积分冻结和转账的事务一致性、WebSocket 实时推送。这些是对系统实现的约束。

功能层面，我们覆盖了注册登录、发布接单、私信聊天、成就徽章、举报审核、管理后台六大模块。非功能性需求方面，重点关注安全性、可维护性、响应式体验、事务一致性和 CI/CD 质量门禁。

## Slide 7 · 体系结构设计 — 杨佳兴

我们的系统采用了经典的分层体系结构，从上到下三层。

展示层基于 Vue 3 加 Vite 5 加 Vant 4，用 Pinia 做状态管理，支持响应式双视图——手机端卡片视图和桌面端表格视图。逻辑层基于 Spring Boot 3.2.5，Spring Security 加 JWT 做无状态鉴权，MyBatis-Plus 做 ORM，WebSocket STOMP 做实时推送。数据层用 MySQL，通过 Flyway 管理从 V1 到 V14 的数据库版本演进。

在架构决策上，我们做了四个关键的 ADR：第一，选择模块化单体而不是微服务——十周的交付周期不需要分布式事务的复杂性，未来如果需要拆分，包结构已经按模块隔离好了。第二，JWT 无状态认证而不是 Session——跨域更灵活，不需要服务端存储会话。第三，单表继承而不是多表继承——用 discriminator 字段加 JSON 属性列，避免了多表 JOIN。第四，悲观锁而不是乐观锁——对于积分这种强一致性要求的场景，SELECT FOR UPDATE 比乐观锁重试更可靠。

---

## Slide 8 · 包结构 — 杨佳兴

这是我们的逻辑包图。后端按照分层架构组织了 10 个 Controller、14 个 Service、17 个 Mapper、16 个 Entity、12 个 DTO 和 7 个 Config 类。

每个包职责清晰：Controller 处理 HTTP 请求和参数校验；Service 封装业务逻辑和事务管理；Mapper 负责数据访问；Entity 映射数据表；DTO 做数据传输；Config 管理安全和 WebSocket 等基础设施配置。大家看这张图就能有一个直观的感受。

---

## Slide 9 · 数据库设计 — 杨佳兴

数据库层面，我们设计了 15 张表，覆盖了四条主线。

第一条是任务流——user、demand、demand_application 支撑需求的发布、接单和状态流转。第二条是积分账本——user_account 加 points_transaction，每个积分变动都有不可变的流水记录，账户余额只能通过 SUM 流水来验算。第三条是聊天和通知——chat_session、chat_message、notification 支撑实时私信和系统通知。第四条是治理和激励——evaluation 做评价，report 做举报审核，badge_definition 和 user_badge 做成就系统，team 和 team_member 做组队协作。

数据库版本通过 Flyway 管理，从 V1 到 V14，每次结构变更都可以追踪、回滚、复现。

## Slide 11 · API 规范 + 设计类图 — 沈诺

## Slide 22 · 白盒测试 — 胡皓轩

白盒测试方面，我们根据代码内部结构设计了三个层级的覆盖标准。

语句覆盖是最基础的——每条代码语句至少执行一次，强度较低，但我们所有方法都达到了这个标准。分支覆盖要求每个判断的真假分支都被覆盖——比如一个 if-else，两个分支都要测到。路径覆盖是最严格的——覆盖主要独立执行路径，对于有循环和嵌套条件的复杂方法，我们重点做了路径覆盖。

实际测试按层级分布：Service 层 135 个用例，用 JUnit 5 加 Mockito，重点测业务规则、事务逻辑和异常路径。Controller 层 102 个用例，用 MockMvc，重点测 REST 参数校验、鉴权和响应格式。WebSocket 层 6 个用例，用 STOMP Test，测连接建立、消息推送和消息格式。

总共 243 个测试用例，全部通过。

---

## Slide 23 · 黑盒测试 — 胡皓轩

黑盒测试方面，我们不看代码，只依据需求规格来设计用例。

以积分奖励的输入为例——有效范围是 1 到 100。我们做了等价类划分：有效等价类 1 到 100 应该允许提交；无效等价类小于 1 或者大于 100 应该提示错误；还有额外的无效类，比如空值、非数字、小数，前端会直接阻止提交。

边界值分析上——最小值减一是 0，拒绝；最小值 1，通过；最小值加一 2，通过；最大值减一 99，通过；最大值 100，通过；最大值加一 101，拒绝。六个边界点覆盖了输入域的全部边界情况。

总结来说，我们的测试策略是有效路径加异常路径加边界路径，三个维度全覆盖。

## Slide 24 · CI/CD 持续集成与部署 — 胡皓轩

我们在 GitHub Actions 上搭建了两条自动化流水线。

后端 CI：push 或者 PR 触发后，自动 Checkout 代码、设置 JDK 17、用 Maven 跑测试和 Checkstyle 代码风格检查。平均执行时间大约一分钟二十秒，测试结果会自动反馈到 PR 页面上。

前端 CI：同样的触发方式，Checkout 后用 npm ci 装依赖，然后 npm run build 构建加 ESLint 检查。平均只要二十四秒，构建结果也是自动反馈。

部署架构上，我们用 Nginx 做统一入口和 HTTPS 终止——所有请求先到 Nginx，以 /api/ 开头的转发到后端 8080 端口，其余转发到前端 5173 端口。这样前后端部署在同一域名下，避免了跨域问题。
