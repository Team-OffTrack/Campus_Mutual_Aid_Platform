### ADR 01：选择分层架构而非微服务架构

**状态：** 已通过
**背景：** 团队需在有限时间内交付包含跑腿、二手交易、社交等功能的复杂平台，团队成员以学生为主。

**决策对比：**
```diff
  我们将采用分层单体架构作为系统基础。
- AI初稿：微服务可以提供更好的扩展性，适合长期发展。
+ 人工修订：鉴于 10 周交付周期及团队运维经验有限，微服务引入的分布式事务和网络治理成本过高。
+ 人工修订：采用“模块化单体”设计，包结构按业务划分（如 com.app.task），为未来剥离微服务预留可能。
```
**后果：** 开发效率最大化，部署简单，但需严格控制各模块间的耦合。

### ADR 02：基于 JWT 的身份认证与学号核验

**状态：** 已通过
**背景：** 系统需要确保用户为校内学生，且需平衡“安全性”与“移动端体验”。

**决策对比：**
```diff
  系统采用 JWT (JSON Web Token) 配合学校教务系统 API 进行认证。
- AI初稿：使用 Session 存储用户信息，简单直接。
+ 人工修订：弃用 Session，改用无状态 JWT。因为平台需支持潜在的跨域调用（如内嵌到学校企业微信），JWT 更具灵活性。
- AI初稿：仅存储学生学号。
+ 人工修订：载荷（Payload）中需包含脱敏后的身份标签（Role），减少业务层频繁查询数据库判断权限的次数。
```
**后果：** 支持横向扩展服务器，减少数据库 I/O 开销，但需处理 Token 续期与黑名单失效逻辑。

### ADR 03：积分托管中间人机制实现

**状态：** 已通过
**背景：** 需求 3 提到需要“平台作为中间人保留酬劳”，以防止跑腿任务中的违约行为。

**决策对比：**
```diff
  引入“冻结积分”逻辑作为信用担保。
- AI初稿：用户点击接单后，由接单者直接向发布者扣款。
+ 人工修订：这会导致严重的资金安全隐患。改为：发布任务时立即从发布者余额扣除并进入“冻结状态”；
+ 人工修订：只有当双方点击“确认完成”后，系统才将冻结积分转入接单者账户。
+ 人工修订：若产生纠纷，引入“管理维护模块”的人工介入接口，支持积分的原路退回。
```
**后果：** 有效解决了“需求 3”中的互助信任问题，增加了数据库事务的复杂性，但保障了资金安全。
