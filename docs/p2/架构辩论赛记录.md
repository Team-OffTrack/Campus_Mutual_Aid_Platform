---

### 方案 A：分层架构 (Layered Architecture)

这是最传统的架构模式，通常分为：**表现层 (UI)、业务逻辑层 (Service)、数据访问层 (DAO)**。对于校园项目，所有功能（跑腿、失物、积分等）都运行在同一个进程中。



*   **优势：** 开发效率极高，调试方便，不需要处理复杂的分布式事务和网络延迟。
*   **劣势：** 代码耦合度高。如果“跑腿服务”的代码写出了 Bug 导致崩溃，整个平台（包括失物招领和积分）都会瘫痪。

### 方案 B：微服务架构 (Microservices Architecture)

将不同功能拆分为独立的服务，例如：用户服务、任务服务（跑腿）、交易服务（二手）、消息服务（社交）。服务之间通过 REST API 或消息队列通讯。



*   **优势：** 模块化极强。后期如果“二手交易”流量激增，可以单独给它增加服务器资源，而不影响其他功能。
*   **劣势：** 基础设施搭建成本极高。你需要处理服务注册、发现、负载均衡以及分布式数据一致性（比如积分扣除成功但任务状态没更新怎么办）。

---

### 架构方案对比表

| 对比维度 | 方案 A：分层架构 (Monolith) | 方案 B：微服务架构 (Microservices) |
| :--- | :--- | :--- |
| **开发复杂度** | **低**：单一工程，逻辑直观。 | **高**：涉及服务拆分、接口定义和分布式通信。 |
| **可维护性** | **中**：初期容易，后期代码量大后会变得臃肿。 | **高**：服务解耦，单个模块的修改不影响全局。 |
| **可扩展性** | **低**：只能对整个应用进行水平扩展。 | **高**：可针对特定高频模块（如跑腿）精准扩容。 |
| **性能** | **高**：本地方法调用，无网络开销。 | **中**：存在跨服务的网络延迟。 |
| **团队学习成本**| **低**：掌握基础 Web 开发即可上手。 | **高**：需学习 Docker, API 网关, RPC 等技术栈。 |
| **10 周内可行性** | **极高**：非常适合快速迭代并上线。 | **低**：光是搭建底层设施可能就要花掉 3-4 周。 |

## 🏛️ 方案确认：为什么“模块化单体”是明智之举

你们选择方案 A 并采用“模块化单体”的思路，完美规避了分布式系统的“三大坑”：网络延迟、分布式事务处理、以及运维自动化成本。



### 1. 应对 10 周交付的“生存法则”
微服务架构的底层设施（网关、服务发现、配置中心、链路追踪）建设往往需要消耗 30% 以上的开发精力。在 10 周的项目中，这几乎等同于“自杀”。单体架构让你们能把 **100% 的火力**集中在核心业务逻辑（如跑腿匹配算法、积分结算安全）上。

### 2. “模块化”是留给未来的后门
你们提到的 `com.platform.task`, `com.platform.user` 包划分非常关键。
> **架构师建议：** 即使在同一个数据库中，也要尽量**避免跨模块的数据库联表查询（Join）**。如果跑腿模块需要用户信息，通过定义的内部 Service 接口获取，而不是直接查询用户表。这样未来如果真的要拆分微服务，只需要把内部接口调用改为 RPC 调用即可。

## 🛠️ 核心功能点的技术实现分析

针对问卷中用户提到的几个痛点，从架构层面我补充几个考量维度：

### 🔐 安全与认证（学号认证）
* **隐私平衡：** 问卷显示用户既想要学号认证的安全性，又想要匿名的自由度。
* **方案建议：** 采用 **JWT (JSON Web Token)** 进行状态管理。后端在认证学号后，允许用户设置一个“展示昵称”。数据库存储真实的学号关联，但对外接口仅展示 UID 或昵称，实现“实名认证，匿名社交”。

### 💰 酬劳与中间人机制（Escrow）
用户提到的“平台作为中间人保留酬劳”是典型的**托管协议（Escrow Pattern）**。
* **流程建议：**
    1.  发单者预扣积分/现金 → 2. 平台锁定该笔资金 → 3. 接单者完成任务 → 4. 双方确认 → 5. 平台释放资金。
* **架构挑战：** 即使是单体架构，也要确保这个过程的**原子性**。必须使用数据库事务（Database Transaction）确保：不能出现“积分扣了，但订单没生成”的情况。

### 💬 交流与反馈
* **即时性需求：** 考虑到 10 周时间，不建议从零开发 WebSocket 聊天服务器。
* **方案建议：**
    * **低配版：** 采用“轮询”或直接接入成熟的第三方 IM SDK（如腾讯云 IM、网易云信）。
    * **折中版：** 仅做任务下方的留言板（类似闲鱼鱼塘），利用推送通知告知用户。

## 💡 架构师的最后嘱托

在单体应用中，最容易腐坏的是**数据模型**。

## 辩论结论

针对我们的校园互助平台，经过激烈讨论，我们最终选择 **方案 A（分层架构）**。

**理由如下：**
1.  **交付压力：** 10 周的时间非常紧迫。微服务带来的“分布式事务”和“服务间治理”会占据大量开发时间，导致核心业务逻辑（如积分结算、评价机制）无法按时完成。
2.  **业务初期规模：** 校园平台的初期并发量通常不会达到必须使用微服务的量级。
3.  **折中方案：** 你们可以采用 **“模块化单体”** 的思路。在方案 A 的基础上，代码层面保持严格的模块化（例如 `com.platform.task`, `com.platform.user`），这样未来如果用户量真的突破万级，再剥离成微服务也会非常轻松。
