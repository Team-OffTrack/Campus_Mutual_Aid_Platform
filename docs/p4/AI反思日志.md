## 2. AI 助力分析（Q1）

AI 在编码中最有价值的场景：
AI 在本阶段最有价值的编码场景是帮助团队快速梳理前后端接口调用链和模块边界。对于用户认证、需求发布、接单、评价、通知、聊天等模块，AI 能够根据项目结构快速生成检查清单，提醒团队重点核对请求字段、响应字段、状态枚举、权限限制和数据库字段之间是否一致。虽然 AI 不能完全替代人工逐文件检查，但它能显著减少遗漏检查项的概率。

AI 在测试生成中最有价值的场景：
AI 在测试生成中最有价值的场景是帮助团队设计集成测试路径，而不是单纯生成单元测试代码。例如，AI 帮助团队把“用户 A 发布需求 → 用户 B 接单 → 完成任务 → 双方评价 → 生成通知 → 更新信用分”拆解成可验证的业务流程，并指出每一步需要检查的数据库状态、接口返回值和权限状态。这类流程型测试更符合本阶段前后端集成的实际需要。

AI 在调试中最有价值的场景：
AI 在调试中最有价值的场景是长日志分析和工程配置问题排查。例如，在 Spring Boot 启动失败时，Maven 最后只显示 `BUILD FAILURE`，但真正原因是 MariaDB 拒绝 `root` 用户通过 JDBC 登录。AI 能从大量堆栈信息中提取关键错误，并指出根因是 MariaDB root 用户通常使用 unix_socket 认证，项目应创建专用数据库用户。对于 keytool 证书 alias 冲突、跨主机访问失败等问题，AI 也能快速给出分层排查步骤。

## 3. AI 误导分析（Q2）

"AI 代码信任度实验"数据摘要：

* AI 直出的测试通过率：约 65%
* 人工审查修复后的测试通过率：约 95%
* 发现的主要问题类型：

  1. AI 能给出正确方向，但部分结论偏泛化，需要人工结合实际代码确认。
  2. AI 对接口字段一致性的判断不能完全直接采信，需要人工逐项核对 DTO、Entity、Controller 和前端 API。
  3. AI 对业务状态流转的建议较完整，但无法单独判断业务规则是否符合团队设计。
  4. AI 生成的命令大多可用，但路径、用户名、数据库名、服务器 IP 等环境变量必须人工替换。
  5. AI 容易先给出通用排查方案，只有在提供完整日志后，定位准确度才明显提高。

"AI 调试对决"实验总结：

* AI 擅长调试的 Bug 类型：
  AI 擅长处理日志中有明确错误信息的工程问题，例如数据库连接失败、认证失败、证书生成失败、端口访问失败、Maven 启动失败等。对于 `Access denied for user 'root'@'localhost'`、`alias already exists`、`Tomcat started on port 8080 (https)` 这类信息明确的问题，AI 能快速提取关键原因并给出可执行修复步骤。

* AI 不擅长调试的 Bug 类型：
  AI 不擅长独立判断强业务语义 Bug，例如订单状态是否允许跳转、评价是否允许重复提交、通知对象是否完全正确、信用分公式是否符合团队业务设定等。这类问题不仅需要代码信息，还需要业务规则、数据库状态和团队设计意图，AI 只能辅助列出检查点，不能替代最终判断。

* AI 修复方案的典型问题：

  1. 有时只给出“可能原因列表”，没有直接定位唯一根因。
  2. 对项目本地路径、数据库账号、服务器 IP 等上下文不敏感，需要人工替换。
  3. 对安全配置问题需要谨慎，例如数据库用户权限、HTTPS 证书、跨域访问、JWT 放行接口等不能盲目照搬。
  4. 对前后端字段一致性检查容易停留在 checklist 层面，不能代替人工逐字段 review。
  5. 对临时开发环境方案和生产环境方案区分不够明显，需要团队自行判断是否适合长期保留。

## 4. Prompt 改进计划（Q3）

上阶段改进措施的验证结果：
上阶段团队开始尝试在 Prompt 中提供更多上下文，包括完整错误日志、执行命令、项目运行环境、配置文件片段和复现步骤。验证结果表明，这种方式明显提高了 AI 的定位准确率。例如，一开始只提供 `BUILD FAILURE` 时，AI 只能判断 Spring Boot 启动失败；当补充 HikariCP、Flyway 和 `Access denied for user 'root'@'localhost'` 日志后，AI 能准确定位到 MariaDB root 认证方式问题，并给出创建专用用户的解决方案。

编码阶段的 Prompt 最佳实践：
编码阶段应尽量避免只说“帮我实现某功能”，而应明确说明模块边界、已有文件、接口路径、请求字段、响应字段、权限要求和数据库表结构。较好的 Prompt 形式是：“基于当前项目的某个 Controller、前端 API 文件和数据库迁移脚本，检查或实现某个功能，并说明涉及哪些字段、哪些权限判断、哪些异常处理。”这样可以减少 AI 生成与项目结构不匹配的代码。

调试阶段的 Prompt 最佳实践：
调试阶段应一次性提供四类信息：第一是完整报错日志，尤其是 `Caused by` 和核心异常；第二是执行命令，例如 `mvn spring-boot:run` 或 `npm run dev`；第三是相关配置文件片段，例如 `application-dev.yml`、Vite 配置、数据库连接配置；第四是已经尝试过的修复方式。这样 AI 能更快排除无关原因，避免反复给出泛泛建议。

下阶段计划改进：
下阶段计划把 AI 主要用于三个方面：第一，生成接口联调 checklist；第二，辅助编写测试用例和异常场景；第三，分析 CI/CD 失败日志。对于 AI 直出的代码，团队计划继续坚持“先人工审查，再本地运行，再通过 CI 检查”的流程，不直接合并未经验证的 AI 代码。同时，团队会在 Prompt 中明确要求 AI 标注“不确定点”和“需要人工确认的地方”，降低误用风险。

## 5. 本阶段的核心工程判断

决策内容：
本阶段最重要的工程判断是：项目后端数据库连接不应继续使用 MariaDB 的 `root` 用户，而应创建 `campus_help` 专用数据库用户，并修改 Spring Boot 的 `application-dev.yml` 数据源配置。同时，在跨主机访问场景下，需要区分后端 HTTPS 服务、前端 Vite 服务、监听地址、防火墙、证书 SAN 和前端代理地址等多个层面，而不能仅凭 ping 通判断服务可访问。

为什么 AI 无法单独做这个决策：
AI 可以根据日志判断 `root` 用户访问被拒绝，也可以建议创建专用数据库用户，但它无法直接知道团队当前数据库中已有多少数据、是否允许重置 MariaDB、是否需要兼容原始 root 配置、服务器真实 IP 是什么、是否有防火墙策略、是否要求生产环境安全规范。也就是说，AI 能提供技术路线，但不能替团队承担环境风险和业务风险判断。

团队最终如何判断和取舍：
团队最终采用了“AI 建议 + 人工确认 + 实际运行验证”的方式。对于数据库问题，团队没有继续强行修改 root 用户认证方式，而是创建专用用户 `campus_help` 并授权 `campus_help` 数据库，这样既解决了 Spring Boot JDBC 登录失败的问题，也降低了使用 root 用户的安全风险。对于证书问题，团队判断如果后端已经能通过 HTTPS 启动，就不重复生成证书；如果需要局域网其他主机访问，再重新生成包含服务器 IP 的 SAN 证书。对于跨主机访问问题，团队按照 AI 给出的排查顺序，依次检查端口监听、访问协议、防火墙、Vite `--host 0.0.0.0` 和前端代理地址，最终将问题拆解为可验证的小问题逐一处理。
