# 校园互助平台 — HCI 演示脚本（精简版）

> 目标：给老师当面展示功能，每个场景讲清楚「做了什么交互 → 对应哪个 HCI 知识点」。控制在 8-10 分钟，挑 ⭐ 场景优先展示。

---

## 场景速查

| # | 路由 | 展示什么 | 核心 HCI 知识点 | 时长 |

## 场景 1：登录 → 首页 — EEC 模型完整闭环 ⭐

**操作**：登录 → 进入首页

**展示**：
1. 输入框 focus 时边框变紫 + 光晕 → **信息丰富的反馈**（Shneiderman 第 3 条）
2. 登录成功 → toast + 自动跳转 → **EEC 评估阶段完整闭环**（系统状态变化即时传达）
3. 首页「早上好☀️/下午好🌈/夜深了🌙」依时间切换 → **用户体验目标**：有温度
4. 签到卡片：未签=紫色按钮 + gift 图标，已签=绿色边框 + 对勾 + 连续天数 → **系统状态可见度**（Nielsen 第 1 条），用户一眼识别今天签没签

**对应笔记**：§3.1 EEC 模型的 7 个阶段（目标→执行→评估）；§4.8 Nielsen 第 1 条"系统状态的可见度"

**话术**：「登录页的焦点反馈让用户知道当前在哪——对应 Shneiderman 黄金规则第 3 条。签到卡片做了两态切换——未签和已签颜色、图标、按钮都不一样，用户不用点进去就知道今天签没签到。这是 Nielsen 启发式规则第 1 条——系统状态可见度的直接应用。」

---

## 场景 2：需求列表 — 响应式切换 + Fitts 定律 ⭐

**操作**：`/demands` → 拖拽浏览器窗口宽度

**展示**：
1. 手机宽度：卡片列表 + 右下角紫色 FAB 大加号 → 拇指热区（**Fitts 定律**：大目标 + 屏幕边缘）
2. 拉宽到 768px+：卡片自动变成数据表格 → **普遍可用性**（Shneiderman 第 2 条）
3. FAB 按钮点击有弹簧回弹（cubic-bezier overshoot）→ 操作被接收的物理隐喻
4. 类型彩色 chip（跑腿=橙/交易=绿…）→ **依赖识别而非记忆**（Nielsen 第 6 条）

**对应笔记**：§8.4 Fitts 定律（MT = a + b×log₂(A/W+1)，目标越大/越近→越快）；§4.7 Shneiderman 第 2 条"符合普遍可用性"

**话术**：「需求列表最典型的 HCI 设计是响应式双视图——手机卡片、电脑表格，同一套代码靠 CSS 媒体查询自动切换。对应 Shneiderman 黄金规则第 2 条——符合普遍可用性。右下角 FAB 按钮用了 Fitts 定律——大目标在拇指热区，点击有弹簧回弹，让用户感知到操作被接收了。」

---

## 场景 3：发布需求 — 弹簧动画 + 防重复提交 ⭐

**操作**：`/demands/publish?type=errand`

**展示**：
1. 类型选择器 6 个 chip → 点击切换有弹簧缩放动画（350ms spring 曲线）→ **EEC 执行阶段反馈**
2. 每个表单字段 focus 时边框变紫 → 用户知道「当前在哪」
3. 提交按钮点击后立即变灰转圈（`:loading`）→ **预防错误**（Shneiderman 第 5 条），防重复提交
4. 提交成功 → toast「发布成功」→ 自动跳转到列表，新需求出现在顶部 → **结束信息**（Shneiderman 第 4 条），用户明确知道任务完成

**对应笔记**：§4.7 Shneiderman 第 4 条"生成结束信息"、第 5 条"预防并处理错误"

**话术**：「发布流程的重点在反馈和防错。类型 chip 用了 spring 弹簧曲线——末尾有 6% 过冲回弹，模拟真实物理手感。提交按钮点一次就变灰转圈，防止用户焦虑连点——对应黄金规则第 5 条预防错误。发布成功后三重反馈——toast + 跳转 + 列表刷新，对应第 4 条『生成结束信息』——让用户确信任务完成了。」

## 场景 4：需求详情 — 确认弹框 + 状态流转 ⭐

**操作**：点击需求 → `/demands/:id`

**展示**：
1. 状态标签颜色语义（OPEN=蓝/进行中=琥珀/完成=绿/取消=红灰）→ **系统状态可见度**
2. 点击「我要接单」→ 弹出确认框「接单后请按时完成需求，确定吗？」→ **预防错误** + **内部控制点**（用户可以取消）
3. 接单后状态标签即时变色 + 出现接单人信息 → **EEC 评估阶段**
4. 评价区：5 星评分点击有缩放动画，提交后可修改 → **让操作容易撤销**（Shneiderman 第 6 条）
5. 管理员看别人需求时举报按钮换成了红色「删除此需求」→ **安全菜单设计**（危险操作与其他操作分开）

**对应笔记**：§4.7 Shneiderman 第 5 条、第 6 条、第 7 条；§4.8 Nielsen 第 1 条

**话术**：「需求详情体现了两个核心 HCI 原则。一是所有不可逆操作都弹确认框——接单、完成、取消，让用户有『再想想』的机会——对应黄金规则第 5 条预防错误和第 7 条内部控制点。二是状态标签颜色语义——四种状态四种颜色，用户扫一眼就知道进展——对应启发式规则第 1 条系统状态可见度。」

## 场景 5：成就徽章 — 游戏化激励 + 全屏动效 ⭐

**操作**：`/badges` → 展示徽章列表 → 触发一个徽章（发布需求/接单）

**展示**：
1. 9 种徽章 3×3 网格，未获得=灰色半透明+进度条，已获得=彩色+「佩戴」按钮 → **状态可见度**
2. 佩戴徽章 → 头像角标即时更新，全局可见 → 社交展示激励
3. 全屏获得动效（`<teleport to="body">`）：徽章卡片从 0 弹到 1.1 再落到 1.0 + 粒子背景 → **峰值时刻设计**（Peak-End Rule）
4. 同一徽章不会重复弹窗（localStorage `earnedBadgeKeys`）→ 不骚扰用户

**对应笔记**：§4.3 用户体验目标（游戏化激励，挑战与成就）；§4.8 Nielsen 第 1 条

**话术**：「徽章系统是游戏化设计在非游戏场景的应用。徽章获得瞬间的全屏动效用了 spring 弹性曲线——这不是炫技，而是心理学上的『峰值时刻』设计，让用户在达成成就时感受到仪式感和喜悦。动效用了 Vue 的 `<teleport>` 组件跨路由渲染，即使获得徽章时用户正在跳转页面，动画也不会被切断。徽章佩戴后以角标形式显示在头像上，全站可见，形成社交展示激励。」

## 场景 6：私信聊天 — 不对称气泡 + WebSocket 实时

**操作**：`/chat/:id`（两个窗口分别登录不同用户）

**展示**：
1. 自己消息=右边紫色渐变+右下小圆角，对方消息=左边白色+左下小圆角 → 视觉层次（**格式塔-图底原则**）
2. WebSocket 实时推送 → 对方发消息这边**零延迟**出现（不再是 10 秒轮询）→ **状态可见度最高标准**
3. Emoji 面板插入光标位置（非末尾）→ 细节体验
4. Enter 发送 → **高效率**（Nielsen 第 7 条"灵活性和高效性"）

**对应笔记**：§3.4 格式塔心理学（图底原则）；§4.8 Nielsen 第 1 条状态可见度（实时性=最高标准）

**话术**：「聊天气泡的不对称圆角用了格式塔心理学的图底原则——自己的消息和别人的消息一眼区分，不需要看头像。WebSocket 替代了之前的 10 秒轮询——消息实时推送，零延迟。这对应启发式规则第 1 条系统状态可见度的最高标准——实时性是状态可见度的极致。」

---

## 场景 7：管理仪表盘 — 信息可视化

**操作**：admin 登录 → 首页有「管理后台」卡片（举报待处理红色角标）→ 进入 `/admin`

**展示**：
1. 4 个统计卡片（用户/需求/积分/举报），数字格式化（1.2k/1.2w）→ **审美和最小化设计**（Nielsen 第 8 条）
2. 举报待处理 >0 时红色高亮 + 角标 → **状态可见度**——需要关注的事项一眼识别
3. 需求类型分布 chips → **信息可视化**（数据图形化 > 纯数字）
4. 3 个快速入口卡片 → 子页面统一 `router.back()` 返回 → **一致性**（Shneiderman 第 1 条）

**对应笔记**：§4.8 Nielsen 第 8 条"审美感和最小化设计"；§4.7 Shneiderman 第 1 条"尽可能保证一致"

**话术**：「管理仪表盘的设计重点是信息架构——4 个统计卡片把关键数据放在首屏，符合 Nielsen 第 8 条审美和最小化设计——不放无关信息。数字做了格式化处理（过千显示 1.2k），减轻认知负担。举报待处理数用红色高亮+角标双重强调——状态可见性原则的直接应用。」

## 涉及的 HCI 知识点汇总

| 知识点 | 课件位置 | 项目对应 |
|--------|----------|----------|
| **EEC 模型**（7 阶段 + 执行/评估隔阂） | §3.1，第 3 讲 | 登录→首页→发布→完成全流程闭环 |
| **格式塔心理学**（图底/相近/连续/相似） | §3.4，第 3 讲 | 不对称气泡、类型色彩分组、对齐布局 |
| **7±2 短时记忆** + 识别优于回忆 | §3.5，第 3 讲 | 功能网格图标+描述、彩色类型标签 |
| **可用性六维度**（易学/易记/效用/效率/安全/满意） | §4.2，第 3 讲 | 全平台覆盖 |
| **Shneiderman 8 条黄金规则** | §4.7，第 3 讲 | 一致性、反馈、防错、撤销、结束信息… |
| **Nielsen 10 条启发式规则** | §4.8，第 3 讲 | 状态可见、现实吻合、识别优于记忆… |
| **Fitts 定律**（MT = a + b×log₂(A/W+1)） | §8.4，第 8 讲 | FAB 大按钮、屏幕边缘利用、右键菜单 |
| **用户体验目标**（质感/温度/Delight） | §4.3，第 3 讲 | 弹簧动画、问候语、全屏徽章动效、彩蛋 |
| **四种评估范型** | §9.4，第 9 讲 | 启发式评估建议（5 专家→80% 问题） |
| **简化设计四策略**（删除/组织/隐藏/转移） | §7.2，第 7 讲 | 管理仪表盘信息架构、表单字段组织 |

## 建议演示顺序

如果只有 **5 分钟**：场景 1→2→4→5→7（登录首页→响应式→确认防错→徽章动效→仪表盘）

如果有 **8-10 分钟**：完整 1→2→3→4→5→6→7

每个场景控制在 1-1.5 分钟：30 秒操作 + 40 秒讲 HCI 对应 + 留白给老师提问。
