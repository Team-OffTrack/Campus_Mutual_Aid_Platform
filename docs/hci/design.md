# 校园互助平台 — 交互设计说明

> 配合人机交互课程线下展示使用。记录项目做了哪些交互设计、对应哪些课件知识点、具体代码在哪里。

## 项目概况

校园互助 Web 应用，Vue 3 + Vant 4 前端。功能覆盖：发布需求（跑腿/交易/组队/失物招领/学习互助）、浏览接单、私信聊天、双向评价、信誉评分。移动端和桌面端自适应。

## 一、可用性六个维度

> 课件第 3 讲：可用性目标（易学性、易记性、效用性、高效率、安全性、主观满意度）

### 1. 好学吗？（易学性 Learnability）

**做了什么**：首页每个功能入口都是「图标 + 名称 + 一句话描述」，不靠文字堆砌，一眼能看懂。表单每个输入框带图标（锁=密码、人=学号）和 placeholder 提示，不需要说明书。

**代码位置**：`Home.vue` 的 `features` 数组（第 152-158 行），Login/Register 的 `field-wrap` + `van-icon` 组合。

**理论上怎么度量**：找几个没用过的同学，统计第一次完成「发布一条需求」需要多长时间。课件提到「10 分钟法则」，我们的目标是 3 分钟内完成首次发布。

### 2. 记住了吗？（易记性 Memorability）

**做了什么**：
- 导航栏、底部按钮、发布按钮位置所有页面都一样 → 肌肉记忆
- 六种需求类型各有固定配色（跑腿=橙、交易=绿、组队=蓝、失物招领=紫、学习=红、其他=灰），列表/详情/发布页全部统一
- 「我发布的」「我接取的」tab 切换，不用去不同页面找

**课件依据**：易记性五因素——有意义的图标 ✓、固定位置 ✓、逻辑分组 ✓、通用惯例 ✓、多通道编码（颜色+文字双通道）✓。

**代码位置**：`TYPE_STYLES` 常量在 `DemandList.vue`、`DemandDetail.vue`、`MyOrders.vue` 三处定义完全相同。`main.css` 的 `--feat-1-bg` 到 `--feat-6-bg` 变量。

### 3. 功能够用吗？（效用性 Utility）

**做了什么**：从发布需求 → 浏览搜索 → 接单 → 私信沟通 → 完成确认 → 互相评价 → 信誉积分，全流程在平台内闭环，不需要切到微信或其他工具。

**课件对比**：课件举例「绘图软件只能画多边形不能徒手画 = 低效用」。我们覆盖了互助流程的全部环节。

### 4. 效率高吗？（高效率 Efficiency）

**做了什么**：
- 手机上：顶部粘着搜索+排序栏，右下角 FAB 大加号按钮一键发布，拇指刚好够到
- 电脑上：自动切换到数据表格视图，信息密度远高于卡片列表
- 首页 6 个快捷入口，一步到最常用功能

**代码位置**：`DemandList.vue` — 移动端 `van-list` 卡片流（第 68 行）+ 桌面端 `demand-table` 表格（第 36 行），通过 `@media (min-width: 768px)` 自动切换（第 293 行）。FAB 按钮（第 97 行）。

### 5. 会误操作吗？（安全性 Safety）

**做了什么**：
- 取消需求、确认接单、确认完成 — 全部弹确认框，不是一点就执行
- 退出登录弹确认框
- 提交按钮点一次后变灰转圈（`:loading`），防止重复提交
- 上传图片检查类型和大小（只允许 image/*，最大 5MB）
- 管理后台不能封禁自己

**代码位置**：`DemandDetail.vue` 的 `handleCancel`/`handleAccept`/`handleComplete`（第 398-423 行）都先调 `showConfirmDialog`。`Home.vue` 和 `Profile.vue` 退出前也弹确认。表单防重复通过 `van-button` 的 `:loading` 属性。图片校验在 `DemandPublish.vue` 的 `beforeReadImage`（第 126 行）。

### 6. 好看好用吗？（主观满意度 Satisfaction）

**做了什么**：
- 毛玻璃卡片有层次感（backdrop-filter: blur）
- 首页根据时间显示不同问候语（早上好☀️ / 夜深了🌙）
- 按钮按下去有弹性回弹（spring 曲线），不是死板的 ease
- Hero 区域紫色渐变，有温度不冷冰
- 圆角不对称（消息气泡左边圆角大右边小、详情卡片上大下小），看起来更自然

**代码位置**：毛玻璃=`main.css` 的 `.glass` 类。问候语=`Home.vue` 的 `greeting` computed。弹簧曲线=`main.css` 的 `--spring-fast-spatial` 等。

**度量方式**：可以用 QUIS 量表（课件第 6 讲，Shneiderman 开发的用户交互满意度问卷，9 级标度）做用户调查。

## 二、EEC 模型 — 用户完成一个操作要经历什么

> 课件第 3 讲核心框架：执行/评估活动周期（Execution/Evaluation Cycle），共 7 个阶段

拿「发布一条跑腿需求」走一遍：

### 执行阶段（用户→系统，阶段 1-4）

| 阶段 | 用户在想什么 | 我们怎么帮 |
|---|---|---|
| 1. 确立目标 | 「我要找人帮拿快递」 | 首页「跑腿代取」入口，图标+描述一眼识别 |
| 2. 形成意图 | 「点发布按钮，填表单」 | 右下角大加号 FAB，拇指范围 |
| 3. 确定动作序列 | 「选类型→写标题→填描述→标报酬→提交」 | 表单自上而下一路到底，步骤线性不绕 |
| 4. 执行动作 | 点击、输入、提交 | 按钮够大（高度 50px），字段有焦点高亮 |

### 评估阶段（系统→用户，阶段 5-7）

| 阶段 | 用户在想什么 | 系统反馈 |
|---|---|---|
| 5. 感知系统状态 | 「提交成功了吗？」 | toast 弹出「发布成功」 |
| 6. 解释系统状态 | 「成功了，跳转了」 | 自动跳转到需求列表，新需求出现在顶部 |
| 7. 评估结果 | 「嗯确实发出去了」 | 列表里能看到刚发的需求 |

### 执行隔阂 & 评估隔阂

课件说 EEC 最大的价值是帮我们发现两个「隔阂」：

- **执行隔阂**：用户想做的事 vs 系统允许做的事。比如用户想改刚发的需求，如果找不到编辑按钮就有隔阂。我们在详情页提供了编辑、取消按钮来缩小这个隔阂。
- **评估隔阂**：系统实际状态 vs 用户预期。比如点了提交按钮后页面没反应，用户不知道成功还是失败。我们用 **toast + 跳转 + 列表刷新** 三重反馈来消除这个隔阂。

## 三、设计规则 — 三层体系

> 课件第 3 讲：Alan Dix 基本规则 → Shneiderman 八条黄金规则 → Nielsen 十条启发式规则

### Alan Dix 三层基本规则

**可学习性**（新用户能否快速上手）：首页功能入口带图标和描述、表单有校验提示、导航结构清晰。

**灵活性**（用户能否以多种方式完成任务）：需求列表支持按类型筛选、关键词搜索、多种排序（最新/报酬高低/即将截止）、手机卡片视图和电脑表格视图。同一个目标多条路径。

**健壮性**（系统对用户的支持程度）：
- 可观察性 → 操作完立即 toast
- 可恢复性 → 需求可取消、评价可修改、封禁可解封
- 响应性 → SPA 架构无白屏等待
- 任务符合性 → 流程天然匹配校园互助场景

### 八条黄金规则逐条对照

| 规则 | 项目做了什么 | 代码位置 |
|---|---|---|
| 1. 尽可能保证一致 | 颜色/圆角/阴影/动画全抽成 CSS 变量，一处改全局生效 | `main.css` 全部 token |
| 2. 符合普遍可用性 | 手机卡片流 + 电脑表格视图，自动切换；键盘焦点可见 | `DemandList.vue` 响应式 CSS、`main.css` `:focus-visible` |
| 3. 提供信息丰富的反馈 | 加载有骨架屏、操作完有 toast、出错有 toast+重试按钮、列表为空有图标+文字 | `SkeletonLoader.vue`、各 view 的 loading/error/empty 状态 |
| 4. 设计对话框生成结束信息 | 提交成功弹 toast 并跳转，让用户明确知道「完成了」 | `DemandPublish.vue` handlePublish |
| 5. 预防并处理错误 | 表单校验、重要操作确认框、按钮防重复提交、图片类型/大小检查 | `DemandDetail.vue` confirm dialogs |
| 6. 让操作容易撤销 | 需求可取消、评价可修改、封禁可解封 | `DemandDetail.vue` handleCancel、评价编辑 |
| 7. 支持内部控制点 | 每个页面有返回按钮、用户可以主动选择下一步 | 所有页面的 `van-nav-bar left-arrow` |
| 8. 减轻短时记忆负担 | 需求类型用彩色标签+文字（不用纯文字）、留白够多不拥挤 | 所有列表页的 type badge |

### 十条启发式规则逐条对照

| 规则 | 做了什么 | 代码细节 |
|---|---|---|
| 1. 系统状态可见度 | 骨架屏加载态、toast 操作反馈、小红点未读标记、错误状态+重试按钮 | `SkeletonLoader` 闪光动画、`NavActions` 铃铛 badge |
| 2. 系统和现实世界吻合 | 全中文、学生日常词汇（「跑腿」不是「物流」）、时间显示「3 分钟前」不是时间戳 | `timeAgo()` 函数在多处使用 |
| 3. 用户控制权和自主权 | 每页有返回箭头、操作可撤销、退出要确认 | `van-nav-bar left-arrow`、confirm dialogs |
| 4. 一致性和标准化 | CSS token 体系保证全局统一 | `main.css` 200+ 行 token 定义 |
| 5. 避免出错 | 必填项校验、图片格式/大小检查、按钮防重复、重要操作确认 | forms 的 `:rules`、`beforeReadImage` |
| 6. 依赖识别而非记忆 | 类型用彩色标签+图标、状态用颜色标记（进行中=绿、取消=红灰） | `typeStyle()` 返回带颜色的 badge |
| 7. 使用的灵活性和高效性 | 新手有引导路径，老手有快捷入口（FAB、搜索）；键盘用户有 `:focus-visible` | FAB 按钮、搜索栏、68 行焦点样式 |
| 8. 审美和最小化设计 | 卡片只放必要信息、留白够多 | 每个卡片仅类型+标题+描述+发布者+时间 |
| 9. 帮助识别/诊断/恢复错误 | 所有 API 调用失败都有 toast 提示（全局拦截器），列表加载失败有重试按钮 | `client.js` 响应拦截器、列表 error state |
| 10. 帮助和文档 | 表单 placeholder 即即时帮助、字段有 icon 提示含义 | 每个输入框的 placeholder 和 icon |

## 四、用户体验目标 — 不只是「能用」

> 课件第 3 讲：用户体验目标 vs 可用性目标，二者可能矛盾

### 追求的具体体验

- **有质感**：毛玻璃 blur + 弹簧动画，按钮按下去有回弹感
- **有温度**：首页依时间段换问候语、Hero 用暖色渐变、头像用彩色首字母
- **令人安心**：重要操作有确认框、操作完有明确反馈、出错有恢复路径
- **不打扰**：动画有目的（反馈操作、引导注意），不是纯炫技

### 可用性 vs 体验的权衡

课件举例：游戏越难越好玩、塑料锤子砸地鼠比鼠标费劲但更有趣。我们的权衡：

- **弹簧动画多花 100-200ms**，比普通 ease-in-out「慢」，但用户感知到操作被接收了，这个感知价值 > 零点几秒
- **毛玻璃效果**在低端机上可能略卡，但校园用户设备主流性能足够

### Dark Pattern

课件提到 Dark Pattern 是指用设计诱导用户做不想做的事。本项目完全避免：确认框的取消按钮不会故意做小或藏起来、不自动勾选额外选项、信誉评分公式公开透明。

## 五、响应式设计 — 手机和电脑不互相迁就

> 黄金规则 #2「符合普遍可用性」+ 启发式规则 #7「灵活性和高效性」

|  | 手机 | 电脑 |
|---|---|---|
| 视图模式 | 卡片列表（手指友好） | 数据表格（眼睛友好） |
| 发布入口 | 右下角 FAB 大加号（拇指区域） | 导航栏加号图标（鼠标操作） |
| 排版 | 一列到底 | 两列 / 最大宽度居中 |
| 切换断点 | 768px 以下 | 768px 以上 |

**代码实现**：CSS `@media (min-width: 768px)` 切换。`DemandList.vue` 的 `.mobile-list` 在桌面端 `display: none`，`.table-wrap` 在手机端 `display: none`。

## 六、动画设计 — 每个动画都有目的

> EEC 模型评估阶段 + 黄金规则 #3「信息丰富的反馈」+ 启发式规则 #1「系统状态可见度」

| 动画 | 目的 | 时长 |
|---|---|---|
| 按钮按压回弹 | 确认操作已被接收（执行→评估桥接） | 350ms 弹簧 |
| 页面切换上移+淡入 | 空间位置感知，过渡不突兀 | 350ms emphasized |
| 骨架屏闪光 | 告知内容加载中，比白屏干等好 | 2s 循环 |
| 未读小红点脉冲 | 吸引注意但不打断 | 2s 循环 |
| Hero 装饰圆圈浮动 | 空间感，不死板 | 4.5-7s 循环 |

**核心曲线**：`cubic-bezier(0.27, 1.06, 0.18, 1.00)` — 弹簧曲线，动画末尾有 6% 过冲回弹，模拟真实弹簧手感。

## 八、错误处理 — 不让用户困惑

> 启发式规则 #1「状态可见度」+ #9「帮助诊断错误」+ 黄金规则 #3「信息反馈」

### 改了什么

之前的代码有大量 `catch { /* skip */ }` — 网络出错时页面毫无反应。

**全局层**：`client.js` 的 axios 响应拦截器，所有 API 错误统一弹 toast（业务错误显示后端消息，网络错误显示「网络错误」，超时/断网显示「无法连接到服务器」）。

**视图层**（本次新增）：
- `DemandList.vue` / `MyOrders.vue`：首次加载失败显示错误图标 + 重试按钮（不用刷新页面）
- `ChatDetail.vue`：发送消息/图片上传失败显示具体提示
- `DemandPublish.vue`：图片上传失败、发布失败显示提示
- `Profile.vue`：头像上传、保存、改密码失败显示提示
- `UserList.vue`：封禁/解封操作失败显示提示

### 错误反馈的分级

参照课件第 7 讲的可用性问题分级：

| 场景 | 反馈方式 | 为什么 |
|---|---|---|
| 列表首次加载失败 | 错误图标 + 重试按钮（持久） | 用户需要知道为什么空白，需要操作出口 |
| 列表翻页加载失败 | toast（短暂） | 已有数据可见，只需提示新数据没加载 |
| 提交表单失败 | toast（短暂） | 表单还在，用户可以修改重试 |
| 发送消息失败 | toast（短暂） | 消息还在输入框，点重发即可 |
| 后台轮询失败 | 静默（不提示） | 每 10 秒一次，提示会骚扰用户 |

## 九、迭代设计过程

> 课件第 4 讲：形成性评估与迭代设计

从 git 提交记录看设计是怎么一步步改出来的：

1. **先做功能**（`3327e75` ~ `9f63df4`）：登录注册、需求发布接取、聊天 → 能用
2. **做响应式布局**（`d97dc2a`）：加桌面端适配 → 手机电脑都能用
3. **做美化**（`2f5f805`）：毛玻璃效果、渐变、阴影、动画 → 好看
4. **做设计系统**（`14068b6`）：颜色/圆角/阴影/动画全抽成 CSS 变量 → 好改
5. **做 HCI 细节打磨**（本次）：错误反馈、焦点可访问性、确认对话框 → 好用又安心

第三步是关键的：把设计决策变成「变量」而不是「写死的值」。评估发现问题 → 改一个变量全局生效，不用翻几十个文件。这就是课件说的 **低成本迭代**。

几个具体的迭代例子（从 git log）：
- `23b1b28`「去掉通知的全标已读按钮」— 评估发现这个按钮不常用，占空间
- `a72237b`「加表情选择器」— 聊天输入太单调
- `d97dc2a`「加桌面端适配」— 电脑看手机版浪费屏幕
- 本次提交「增强错误反馈和键盘可访问性」— 沉默失败 + 无法用键盘 = 可用性问题

## 十、如果用课件方法评估本项目

> 综合课件第 4-7 讲

### 形成性评估（开发中做，边做边改）

我们实际做的就是简化版形成性评估：改一版 → 自己试用 → 发现问题 → 改。课件推荐的「边做边说」（think-aloud）是最合适的下一步——找个同学，让他对着系统执行任务，嘴里说出在想什么。

### 启发式评估（找 3-5 人对照十条规则检查）

课件第 6 讲关键数据：5 名专家能发现约 80% 的可用性问题（公式 N(1-(1-L)^n)，L≈31%）。找 3-5 个同学对照十条启发式规则逐条检查，每人独立做然后汇总。

### 可用性测试（DECIDE 框架）

课件第 7 讲六步框架：

1. **确定目标**：新用户 3 分钟内能完成首次发布吗？
2. **发掘问题**：哪个步骤最困惑？哪里最慢？
3. **选择方法**：可用性测试 + 边做边说
4. **实际问题**：找 5-8 个同学，实验室环境，1 小时内
5. **道德问题**：签知情同意书，不拍脸只录屏
6. **分析数据**：统计任务完成时间、错误次数、成功率，问题分 0-4 级严重度

## 十一、关键设计决策一览

| 决策 | 做了什么 | 为什么 |
|---|---|---|
| CSS 变量设计系统 | 200+ 行 token 在一个文件 | 一致性（黄金#1）+ 低成本迭代 |
| 全局 axios 拦截器 | 所有 API 错误统一弹 toast | 反馈（黄金#3）+ 省代码 |
| 弹簧缓动曲线 | `cubic-bezier(0.27,1.06,0.18,1.0)` | 用户体验目标：质感 |
| 不对称圆角 | 消息气泡左边≠右边圆角 | 视觉自然，Material Design 3 特征 |
| 响应式双视图 | 手机卡片 / 电脑表格 | 普遍可用性（黄金#2） |
| 焦点可见样式 | 68 行 `:focus-visible` | 键盘可访问（启发#7） |
| 操作确认弹框 | 取消/接单/完成/退出全弹确认 | 防错（黄金#5 + 启发#5） |
| 列表错误+重试 | 加载失败显示图标+重试按钮 | 状态可见（启发#1）+ 帮助恢复（启发#9） |

## 十二、总结

一句话：这个项目不是随便堆了几个页面，而是 **每做一个交互决策都有 HCI 理论支撑**。

- 可用性六个维度全部覆盖
- EEC 模型贯穿交互流程
- 三层设计规则（Dix + Shneiderman + Nielsen）逐条有对应
- 用户体验超越「能用」，追求质感、温度、安心
- 错误处理从沉默失败变成明确反馈
- 键盘用户不被忽略
- 设计系统支持低成本迭代
